# GIM 替换注意事项

## 块头部 offset+8 字段语义（已修复）

通过逆向游戏 EBOOT（`NPJH50107_gvsgnextpsp.BIN`）中的 GIM 块遍历函数 `sub_8805280`，确认 GIM 块头部各字段含义：

| 偏移 | 含义 |
|------|------|
| +0x00 | u16 block_type |
| +0x02 | u16 flags |
| +0x04 | u32 block_size（总大小，含子块） |
| +0x08 | u32 **子块起始偏移**（容器块）/ **block_size**（叶子块） |
| +0x0C | u32 data_offset |

游戏的块遍历使用 `block + *(block+8)` 作为子块迭代起点，`block + *(block+4)` 作为结束边界。因此：

- **容器块**（root 0x02, picture 0x03）：offset+8 必须等于 data_offset（通常为 0x10），否则游戏找不到子块，返回 NULL 导致崩溃。
- **叶子块**（image 0x04, palette 0x05）：offset+8 等于 block_size（无子块可迭代；同时游戏通过 `block + *(block+8)` 导航到下一个兄弟块来定位 palette）。

原始游戏 GIM 文件验证了这一结论（容器块 offset+8 = 0x10）。

## 游戏支持的像素格式

游戏的纹理绑定函数 `sub_89BE7D4` 动态读取 GIM entry 的 pixel_format 设置 GE 命令，并仅对 indexed 格式（format >= 4）加载 CLUT。以下格式均可正确解析和渲染：

| Format ID | 名称 | BPP | 游戏支持 |
|-----------|------|-----|----------|
| 0 | RGBA5650 | 16 | 是 |
| 1 | RGBA5551 | 16 | 是 |
| 2 | RGBA4444 | 16 | 是 |
| 3 | RGBA8888 | 32 | 是 |
| 4 | Indexed4 | 4 | 是 |
| 5 | Indexed8 | 8 | 是（原始格式） |

## PZZ 流大小限制（重要）

**RGBA8888 格式本身不会导致崩溃，但替换后的流数据过大会导致游戏崩溃。**

实测结果（原始纹理为 256×256 Indexed8，约 64KB）：

| 替换格式 | 替换分辨率 | 流大小 | 游戏结果 |
|----------|-----------|--------|----------|
| Indexed8 | 256×256 | ~64 KB | 正常 |
| RGBA4444 | 1024×1024 | ~2 MB | 正常 |
| RGBA8888 | 256×256 | ~256 KB | 正常 |
| RGBA8888 | 1024×1024 | ~4 MB | **崩溃** |

结论：游戏的 PZZ 加载器或内存分配器对单个流的大小存在上限（约 2-3 MB 之间）。超过该上限的流会导致缓冲区溢出或内存不足，引发崩溃。

### 建议

- 使用 **Auto (RGBA4444)** 作为默认替换格式，16-bit 色深对 PSP 游戏纹理足够。
- 如需使用 RGBA8888，应将替换图片缩放到与原始纹理相同或接近的分辨率，避免流数据过大。
- 不同 PZZ 文件的流大小上限可能不同（取决于游戏的内存布局和缓冲区分配），需要实际测试确认。

## PMF2 模型面数/顶点限制分析

### 主 Display List 缓冲区溢出 → 游戏死循环（最关键）

游戏帧初始化函数 `sub_89BD000` 中存在硬编码的缓冲区溢出检查：

```
0x89BD030: lw $v0, dword_8DB9D8C      ; 加载溢出检查启用标志
0x89BD034: beqz $v0, loc_89BD068      ; 首帧跳过检查
0x89BD04C: lui $v0, 4                 ; v0 = 0x40000 (256 KB)
0x89BD050: subu $v1, $a0, $v1         ; 已用字节 = 写指针 - 基地址
0x89BD054: sltu $v0, $v1, $v0         ; 是否 < 256KB?
0x89BD058: bnez $v0, loc_89BD068      ; 未溢出则继续
0x89BD060: b loc_89BD060              ; *** 无限循环（while(1)）= 游戏卡死 ***
```

- **缓冲区大小**：0x40000（256 KB），双缓冲（buffer 0 从 `0x8DF9E80`，buffer 1 从 `0x8E39E80`）
- **溢出后果**：进入 `while(1)` 无限循环，游戏永久卡死
- **缓冲区用途**：每帧所有渲染指令的总缓冲区（所有模型 + 特效 + UI）

### 每个 Section 的渲染开销

每个被渲染的 PMF2 section 会在主缓冲区写入：

| 操作 | 函数 | 主缓冲区消耗 |
|------|------|-------------|
| 纹理绑定 | `sub_89BE7D4` | ~48 字节（12 个 GE 命令） |
| 骨骼矩阵 | `sub_89BEC88` | ~52 字节（13 个 GE 命令） |
| GE CALL | `sub_89BED1C` | ~12 字节（BASE + OFFSET_ADDR + CALL） |
| **合计** | | **~112 字节/section** |

Section 的 display list 本身通过 GE CALL 机制执行（不复制到主缓冲区），但主缓冲区 256 KB 的 CALL 命令开销限制了每帧可渲染的 section 总数。

理论上限：256 KB / 112 B ≈ 2340 sections/帧（实际远小于此，因为场景中还有特效、UI 等其他渲染命令）。

### Section 计数限制

`sub_89945A4` 中渲染循环使用 `char`（有符号字节）存储 section 数量：
```c
// a1 + 1049: section count, type = char (signed byte, max 127)
v4 = 1;
while (v4 < *(char*)(a1 + 1049)) {
    sub_89AD77C(a1, v4 + 128, ...);
    ++v4;
}
```

**每个模型最多 127 个 section。**

### 无硬编码的面数/顶点数上限

游戏代码中没有对单个 section 的面数或顶点数进行验证或限制。PSP GE 的 PRIM 命令使用 16 位顶点计数（最大 65535），我们的代码已用 `0xFFFC`。

### 性能瓶颈（卡顿原因）

PSP 图形引擎（GE）的处理能力有限：
- 顶点变换速率：约 3-5M 顶点/秒
- 30fps 下每帧预算：约 **100K-170K 顶点**（~33K-56K 三角面）
- 此预算为整个场景共享（所有角色 + 特效 + 背景）

原版角色模型一般在 1000-3000 面。**单个模型建议控制在 3000-5000 面以内。**

### 金手指修复 Display List 溢出死循环

将 `0x089BD060` 处的无限循环指令 `b loc_89BD060` 替换为 `nop`，使缓冲区溢出时跳过死循环继续执行：

```ini
_C1 PMF2 Modding - Skip Display List Overflow Hang
_L 0x211BD060 0x00000000
```

效果：溢出时不再卡死，可能出现一帧画面闪烁/错误（因为溢出的 GE 命令被截断），下一帧缓冲区自动重置恢复正常。

**注意**：此金手指仅解决死循环问题，不能解决面数过多导致的卡顿（这是 PSP 硬件性能限制，无法通过金手指解决）。
